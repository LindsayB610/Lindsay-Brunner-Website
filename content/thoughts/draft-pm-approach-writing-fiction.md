First fiction work in roughly twenty years. The approach is shaped more by project management than by prior writing experience: treating narrative as a system to manage, not a mood to access.

## Current working pattern (chapter 2 rebuild)

Chapter 1 changed in ways that matter, so chapter 2 fell out of alignment. The response is not a full rewrite from scratch but a structured rebuild.

### Document structure

- **Top:** active redraft (current truth)
- **Divider:** explicit “legacy” line
- **Bottom:** old draft (raw material, extraction layer)

### Execution model

The outline lives above the legacy line. Each outline item:

- has a purpose
- has a rough word range
- acts as an execution unit

For each section: pull from legacy if valid, rewrite if partially valid, replace if invalid.

---

## Unit of work

Markdown-based workflow. Work happens in roughly fifty-line blocks (literal, not metaphor).

Each block:

- must “compile” against updated chapter 1
- is evaluated for continuity, character motivation, and knowledge consistency

Not evaluated for prose quality, polish, or style improvements. In this phase, correctness over quality.

---

## System characteristics

Legacy text stays until proven unnecessary. No premature deletion (avoids losing usable material). No full rewrites (avoids scope explosion).

This is effectively incremental migration, not drafting.

---

## Mental model

Not “writing chapter 2.” Refactoring downstream dependencies.

Equivalent to updating a system after upstream API changes:

- Chapter 1 = upstream contract
- Chapter 2 = dependent system that needs an update

---

## Key constraint

Do not improve while fixing. Strict separation: alignment phase is correctness only; craft and quality come later. That avoids an accidental rewrite spiral and keeps refactor separate from polish.

---

## Outline role

Not inspiration. A constraint and sequencing system.

It provides ordering, scope boundaries, and a definition of done per segment. It reduces decision fatigue and the blank-page problem.

---

## Observed effects

Above-the-line text tends to be tighter, more intentional, and higher signal density. Below-the-line text is more exploratory, with more drift and “thinking on the page” artifacts.

The refactor process implicitly upgrades quality while aligning structure.

---

## Risks

Over-reliance on outline pacing. Potential to constrain organic expansion if followed too rigidly. (Currently mitigated by letting prose breathe within segments.)

---

## Framing (for a future post)

Potential angle: I did not return to writing as a writer; I approached it as a project manager.

Key contrast: writing as inspiration versus writing as system; mood versus execution model.

---

## Status

The system is working so far. Chapter 2 is in active rebuild with this method. Unknown whether the approach holds at full manuscript scale. Capture observations during the process for a later postmortem.
